# 第五章《模型协同复用优化系统设计与实现》写作辅助文档

## 1. 文档定位

本文档用于辅助撰写学位论文第五章，内容以 `demo_system/` 目录下的现有实现为依据，重点描述系统的工程化设计，而非重复第三章和第四章中的算法原理推导。写作时建议将本章定位为“面向算法展示与验证的系统实现章节”，突出以下三点：

- 如何将第三章的 ModularEvo 方法与第四章的 AutoRouter 方法封装为统一的可视化演示系统。
- 如何通过前后端分离、算法适配层、本地资源管理等方式实现研究代码到系统原型的工程转化。
- 如何利用统一的页面结构、接口设计和数据组织方式支撑模型协同复用优化流程的展示与验证。

本章宜少讲算法公式推导，多讲“系统为什么这样分层、代码如何组织、模块之间如何协同”。

## 2. 建议的章节组织方式

建议第五章按如下结构展开：

### 5.1 系统需求分析

建议从以下几个角度写：

- 展示性需求：系统需要可视化展示模型模块化、模型合并、Router 路由和评测结果。
- 交互性需求：系统需要支持模块加载、参数设置、数据集上传、评测触发和结果回显。
- 复用性需求：系统不能直接改写原始实验代码，而应通过适配层进行封装，降低耦合。
- 可部署性需求：系统应支持本地一键启动，依赖尽量收敛在 Python、Node.js 和本地模型资源上。

### 5.2 系统总体架构设计

这一节重点介绍四层结构：前端展示层、后端接口层、算法适配层、数据与资源层。

### 5.3 前端可视化模块设计与实现

这一节可围绕 Vue 组件划分、页面职责、图谱与日志组件复用、长时任务可视反馈进行说明。

### 5.4 后端服务与算法适配层设计与实现

这一节可围绕 FastAPI 路由设计、Pydantic 数据模型、算法接口封装、资源缓存策略展开。

### 5.5 系统部署与运行机制

这一节建议说明本地模型资源组织、启动脚本、端口映射、日志与 PID 管理。

### 5.6 本章小结

总结系统如何为前文提出的模型协同复用方法提供统一的验证载体和交互式展示平台。

## 3. 基于现有代码的总体架构

当前系统采用典型的前后端分离结构，并在后端之下额外设置算法适配层，使研究代码与 Web 服务逻辑保持解耦。对应关系如下：

```text
用户浏览器
    |
    v
Vue 3 + Vite 前端展示层
    |
    | HTTP / JSON
    v
FastAPI 接口层
    |
    | Python 调用
    v
算法适配层（chapter3 / chapter4）
    |
    | 读取本地模型、数据集、任务向量、Router checkpoint
    v
本地资源层（data/ + SQLite）
```

从工程实现角度看，这种分层有三个直接好处：

- 前端只关心页面状态、参数输入与结果展示，不直接接触模型细节。
- 后端只负责接口编排、输入校验和结果组织，不承担底层算法实现。
- 算法适配层负责复用已有研究代码与本地资源，便于后续替换模型路径、扩展任务或迁移实验数据。

## 6. 前端系统设计与实现

### 6.1 总体布局设计

前端总布局由 `App.vue` 统一管理，采用固定顶部导航栏、固定左侧边栏、右侧主内容区的后台式结构。其实现目标并非通用门户，而是为长流程算法演示提供稳定的信息承载框架。

可以在论文中概括为：

“系统前端采用组件化单页架构，将导航、系统状态、算法展示和结果可视化解耦为多个独立组件，在保证交互统一性的同时，支持不同算法页面复用同一套布局骨架。”

### 6.2 核心页面划分

| 页面 | 路由 | 对应源码 | 功能定位 |
| --- | --- | --- | --- |
| 首页 | `/` | `frontend/src/views/HomeView.vue` | 展示平台简介、两类算法入口卡片 |
| ModularEvo 页面 | `/modularevo` | `frontend/src/views/ModularEvoView.vue` | 展示第三章模块化进化流程 |
| AutoRouter 页面 | `/autorouter` | `frontend/src/views/AutoRouterView.vue` | 展示第四章动态路由合并流程 |

其中首页并不承担复杂业务逻辑，主要负责对两套算法演示入口进行说明与跳转。真正的系统核心在后两个页面中完成。

### 6.3 通用组件设计

| 组件 | 源码 | 作用 |
| --- | --- | --- |
| `AppHeader` | `frontend/src/components/AppHeader.vue` | 顶部导航、路由切换、系统状态标识 |
| `AppSidebar` | `frontend/src/components/AppSidebar.vue` | 轮询后端系统状态，展示 GPU、内存、CPU、磁盘、Python 版本等信息 |
| `KnowledgeGraph` | `frontend/src/components/KnowledgeGraph.vue` | 基于 ECharts 的通用知识图谱组件 |
| `TerminalLog` | `frontend/src/components/TerminalLog.vue` | 模拟终端输出，增强长耗时任务的过程可见性 |

从设计上看，`KnowledgeGraph` 和 `TerminalLog` 是两个最有代表性的复用组件：

- `KnowledgeGraph` 将图谱节点和边作为输入属性，实现第三章与第四章图谱的统一展示。
- `TerminalLog` 将日志行数组作为输入，使模型加载、参数合并、评测执行等过程以终端风格输出，减轻用户等待时的“黑盒感”。

### 6.4 第三章页面的交互链路设计

`ModularEvoView.vue` 将第三章演示流程拆分为六个功能卡片：

1. 算法介绍。
2. 知识图谱展示。
3. 模型家族树与模型详情展示。
4. 稀疏模块加载与层级统计展示。
5. 微调模型评测结果展示。
6. 合并方法选择、参数配置、日志回显与结果对比。

这一设计对应第三章方法“模块化 -> 独立进化 -> 知识组合”的自然流程，便于论文中强调界面结构与算法流程的一致性。

页面初始化时通过 `Promise.all` 并行调用模块列表、微调模型列表和图谱接口，减少首屏等待时间。合并操作则先触发真实后端请求，再在前端模拟分阶段日志，最后将后端返回的真实评测结果回填到图表和日志区域。这样既保证了交互反馈的连续性，也保留了最终结果的真实性。

### 6.5 第四章页面的交互链路设计

`AutoRouterView.vue` 对应的流程为：上传混合数据集 -> 解析任务分布 -> Router 推理 -> 动态合并 -> 分任务评测 -> 归一化结果展示。

页面核心特征有三点：

- 通过上传 CSV/JSON 文件将第四章方法从离线实验流程转化为在线演示流程。
- 使用两个柱状图分别展示 Router 学得的 α 系数以及合并模型相对基线的归一化性能。
- 使用终端日志模拟资源加载、Router 编码、任务向量合并与分类头推理等步骤，使算法内部执行过程更加可解释。

### 6.6 前端通信机制设计

前端接口层采用两级封装：

- `frontend/src/api/index.js` 负责 Axios 实例创建。
- `frontend/src/api/chapter3.js` 与 `frontend/src/api/chapter4.js` 负责业务接口拆分。

这一做法使页面组件只面向语义化函数，如 `getModules()`、`mergeModels()`、`uploadDataset()`、`evaluate()`，而不直接拼接 URL。写作时可将其表述为“通过接口封装降低视图层与后端协议之间的耦合度”。

同时，`vite.config.js` 中将 `/api` 代理到 `http://localhost:8000`，说明系统在开发态采用前后端分离、统一域名前缀访问的方式组织请求。

## 7. 后端服务设计与实现

### 7.1 后端入口与路由组织

`backend/main.py` 负责三项核心工作：

- 初始化 FastAPI 应用和基础元信息。
- 注册 CORS 中间件。
- 在启动阶段调用 `init_db()` 创建数据库表，并挂载第三章、第四章和系统状态接口。

路由层按业务域划分为三部分：

| 路由前缀 | 作用 |
| --- | --- |
| `/api/ch3` | 第三章相关接口 |
| `/api/ch4` | 第四章相关接口 |
| `/api/system` | 系统监控接口 |

这种分组方式有利于论文中说明“按章节算法域拆分 API”，对应关系清晰，后期扩展新的章节或算法时也便于继续横向扩展。

### 7.2 第三章接口设计

第三章后端接口集中在 `backend/api/chapter3.py` 中，主要包括：

| 接口 | 功能 |
| --- | --- |
| `GET /api/ch3/modules` | 返回可用语言模块列表 |
| `POST /api/ch3/load-module` | 加载指定语言稀疏模块并返回 WRR、稀疏率、层统计 |
| `GET /api/ch3/finetuned` | 返回已准备好的微调模型信息 |
| `POST /api/ch3/evaluate/{task}` | 返回预计算评测结果，并模拟评测耗时 |
| `POST /api/ch3/merge` | 触发模型合并与双任务评测 |
| `GET /api/ch3/graph` | 返回知识图谱节点和边数据 |

需要强调的是，该部分接口并未将算法细节硬编码到页面，而是通过统一的输入输出格式对前端暴露服务能力，从而实现页面展示逻辑与底层模型计算逻辑的解耦。

### 7.3 第四章接口设计

第四章后端接口集中在 `backend/api/chapter4.py` 中，主要包括：

| 接口 | 功能 |
| --- | --- |
| `GET /api/ch4/status` | 返回模型资源是否已加载及其概要信息 |
| `GET /api/ch4/baseline` | 返回四个任务的基线准确率 |
| `GET /api/ch4/models` | 返回模型家族信息 |
| `POST /api/ch4/upload-dataset` | 上传并解析用户数据集 |
| `GET /api/ch4/sample-dataset` | 下载预置示例数据 |
| `POST /api/ch4/evaluate` | 执行 Router 推理、动态合并与分组评测 |
| `GET /api/ch4/graph` | 返回第四章知识图谱数据 |

第四章接口的工程重点在于统一输入数据格式。后端要求上传记录遵循 `text + label + task_id` 结构，这使不同任务的数据可以在单一接口中完成解析、分发和评测，有利于论文中突出“异构任务统一建模接口”的设计思想。

### 7.4 系统状态接口设计

`backend/api/system.py` 提供 GPU、CPU、内存、磁盘和 Python 版本的监控能力。其实现方式较轻量：

- 通过 `nvidia-smi` 获取 GPU 名称、利用率、显存和温度。
- 通过 `/proc/meminfo` 获取系统内存总量与已用内存。
- 通过 `shutil.disk_usage` 获取磁盘使用情况。

该设计说明系统虽然不是完整的运维平台，但已经具备基础的运行时可观测能力，可为“算法演示系统”的稳定运行提供必要支撑。

### 7.5 数据模型与契约化设计

后端的数据约束主要体现为两类：

- Pydantic 请求/响应模型：保证接口输入输出字段结构稳定。
- SQLAlchemy 数据表：为任务记录、模型记录、评测结果等信息提供基础持久化能力。

当前数据库中定义了 `Task`、`Model`、`Evaluation` 三张表，说明系统在工程上预留了任务跟踪与结果存储能力。论文撰写时可以表述为“系统已完成基础持久化结构设计，当前原型阶段仍以本地文件资源和运行时结果回传为主”。

## 8. 算法适配层设计

### 8.1 设置算法适配层的原因

如果直接在 FastAPI 路由中调用原始实验脚本，容易带来三个问题：

- 路由层逻辑与算法细节耦合严重。
- 模型路径、数据路径、设备选择等环境信息分散，难以维护。
- 研究代码不易复用到 Web 系统之外的其他场景。

因此，系统在 `algorithm/` 下设置章节级适配层，将“实验代码复用”与“服务接口封装”之间增加一层中介。这是本系统在工程设计上最值得强调的部分之一。

### 8.2 第三章适配层设计

第三章适配层以 `algorithm/chapter3/` 为核心，职责分工如下：

- `config.py`：维护 CodeBERT 基座模型、模块文件、微调模型、评测集和小样本评测集路径。
- `model_loader.py`：提供 `load_base_model()`、`load_sparse_module()`、`load_finetuned_model()` 等统一加载入口。
- `merger.py`：提供 `merge_models()`、`apply_merged_params()`、`merge_and_evaluate()` 等统一合并与评测流程。

其设计特点主要有两点：

- 通过统一函数接口屏蔽底层模型构造细节，使后端路由只需传入任务名、语言名和合并参数。
- 通过排除任务头参数、仅合并编码器参数的方式，使多任务合并逻辑更加稳定，符合第三章实验实现需求。

### 8.3 第四章适配层设计

第四章适配层以 `algorithm/chapter4/adapter.py` 为核心，统一封装了以下能力：

- 模型资源加载：GPT-Neo、任务向量、分类头、Router checkpoint 和 tokenizer。
- 上传数据解析：支持 CSV 和 JSON 两类输入。
- Router 推理：根据输入文本集合估计任务分布并输出 α 系数。
- 动态合并评测：根据 α 对任务向量加权求和，构造合并参数，并在各任务分组上执行评测。

特别值得写入论文的是，这一实现并不是对整份数据只生成一次全局权重，而是提供了 `infer_and_evaluate_per_group()` 这样的分组评测策略：先预测每条样本所属任务，再对每个预测组分别计算 α、分别合并、分别评测，最后汇总整体指标。这说明系统已经将第四章方法中的“按输入分布动态决定模型组合”落到了可执行代码层面。

### 8.4 资源缓存与路径隔离策略

系统使用两类轻量工程策略来提升稳定性：

- 延迟加载与进程内缓存：第三章和第四章都使用 `_cache` 保存已加载模型与中间资源，避免重复加载大型模型文件。
- 路径配置集中管理：模型、数据集、任务向量、Router checkpoint 等路径统一写入 `config.py`，降低路径散落在业务代码中的问题。

另外，后端 `core/config.py` 通过 `DEMO_PROJECT_ROOT` 和 `DEMO_ALGO_ROOT` 环境变量管理系统根目录与算法根目录，为本地部署与路径迁移提供了更好的可配置性。

## 9. 数据组织与运行流程设计

### 9.1 数据组织方式

系统的数据组织采用“本地资源优先”的方式，适合论文原型系统：

- 基座模型、模块文件、任务向量、分类头、Router checkpoint 保存在本地目录。
- 第三章和第四章的演示数据集保存在 `data/datasets/`。
- 用户上传的混合数据保存在 `data/uploads/`。
- 任务、模型和评测记录具备 SQLite 存储基础，但当前系统以文件资源和内存结果回传为主。

这一设计有利于实验复现与演示部署，不依赖外部对象存储或远程模型服务。

### 9.2 系统启动流程

`start.sh` 的实现可以作为论文“部署与运行设计”小节的依据，其流程包括：

1. 设置 `DEMO_PROJECT_ROOT` 与 `DEMO_ALGO_ROOT` 环境变量。
2. 检测 Node.js 可执行文件路径。
3. 检查 8000 和 3000 端口是否被占用。
4. 启动 FastAPI 后端，并将日志写入 `.backend.log`。
5. 轮询健康检查接口，确认后端已就绪。
6. 启动 Vite 前端，并将日志写入 `.frontend.log`。
7. 将前后端进程 PID 写入隐藏文件，便于后续停止服务。

与之对应，`stop.sh` 负责根据 PID 文件停止服务，若 PID 文件不存在，则回退到 `pkill` 方式清理进程。该设计对于论文原型系统而言已经足够完整。

### 9.3 典型业务流程

#### 第三章演示流程

1. 前端页面初始化，请求模块列表、微调模型和图谱数据。
2. 用户选择语言模块并触发加载。
3. 后端调用适配层将稀疏模块应用到 CodeBERT 基座，返回 WRR 和层级统计。
4. 用户触发单任务评测或合并评测。
5. 前端展示终端日志、评测指标和多方法对比图。

#### 第四章演示流程

1. 用户上传符合统一格式的数据集。
2. 后端解析记录并统计任务分布。
3. 前端触发评测接口。
4. 后端加载 GPT-Neo、任务向量、Router 和分类头。
5. 后端先进行任务识别，再按预测分组计算 α、执行合并并评测。
6. 前端展示 α 系数、归一化准确率和终端日志。
